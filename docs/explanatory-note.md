# Пояснительная записка

## 1. Название проекта
Quizzy — веб-приложение для проведения викторин в реальном времени.

## 2. Цель проекта
Цель проекта — разработать работоспособный MVP веб-приложения, которое позволяет:
- регистрировать и авторизовать игроков и организаторов;
- создавать и редактировать викторины;
- запускать квиз в отдельной комнате по коду доступа;
- подключать участников к активной игре в реальном времени;
- показывать результаты в виде лидерборда;
- сохранять историю участия и проведённых игр.

Проект выполнен в соответствии с ТЗ на разработку веб-приложения для проведения квизов.

## 3. Обоснование выбора технологий
Для реализации проекта был выбран стек JavaScript/TypeScript, так как он позволяет использовать единый язык как на клиентской, так и на серверной части.

Использованные технологии:
- `React + Vite + TypeScript` — для создания современного и быстрого пользовательского интерфейса;
- `Node.js + Express` — для серверной логики и REST API;
- `PostgreSQL` — для хранения пользователей, квизов, комнат, вопросов, ответов и истории;
- `Prisma ORM` — для описания модели данных, миграций и удобной работы с базой данных;
- `Socket.IO` — для синхронизации игры в реальном времени между организатором и участниками;
- `Docker Compose` — для контейнеризации проекта и упрощённого запуска на любой машине;
- `Multer` — для загрузки изображений к вопросам.

Выбор PostgreSQL и Prisma позволил быстро перейти от прототипа к устойчивой модели хранения данных. Socket.IO был выбран как наиболее удобный инструмент для реализации синхронной игровой логики с серверным контролем таймеров и ответов.

## 4. Основные этапы разработки
### 4.1. Проектирование структуры приложения
На первом этапе была определена архитектура клиент-серверного приложения. Было принято решение разделить проект на два независимых модуля:
- `client` — интерфейс пользователя;
- `server` — API, игровая логика и работа с базой данных.

Также была спроектирована модель базы данных, включающая пользователей, викторины, вопросы, варианты ответов, игровые комнаты, участников комнат и ответы игроков.

### 4.2. Разработка клиентской части
На клиентской части были реализованы:
- публичная главная страница;
- регистрация и авторизация;
- кабинет игрока;
- кабинет организатора;
- создание и редактирование квиза;
- экран комнаты ожидания;
- экран прохождения викторины;
- экран итоговых результатов.

Интерфейс полностью переведён на русский язык и оформлен в единой визуальной системе.

### 4.3. Разработка серверной части
На серверной стороне были реализованы:
- регистрация и вход пользователей;
- выдача JWT-токена;
- проверка прав доступа для игрока и организатора;
- CRUD-операции для викторин;
- загрузка изображений для вопросов;
- создание игровых комнат;
- получение данных для кабинетов игрока и организатора.

### 4.4. Реализация логики реального времени
Для поддержки реального времени был реализован Socket.IO-сервер. Через него происходят:
- вход организатора и игрока в комнату;
- синхронизация состава участников;
- запуск вопроса;
- передача состояния активного вопроса;
- отправка ответов игроками;
- обновление лидерборда;
- завершение комнаты и фиксация результатов.

Сервер является единственным источником истины. Именно на сервере хранятся текущий вопрос, время окончания раунда, начисленные баллы и итоговые результаты. Это исключает ошибки, связанные с локальным состоянием браузера.

### 4.5. Контейнеризация
Для удобства запуска был подготовлен `docker-compose.yml`, который поднимает:
- `postgres` — базу данных;
- `server` — backend;
- `client` — frontend.

При первом запуске автоматически применяются миграции Prisma и добавляются демонстрационные данные. Благодаря этому любой пользователь может запустить проект локально одной командой.

## 5. Модель данных
В проекте используются следующие основные сущности:
- `User` — пользователь системы;
- `Quiz` — викторина;
- `Question` — вопрос викторины;
- `AnswerOption` — вариант ответа;
- `GameRoom` — игровая комната;
- `RoomParticipant` — участник комнаты;
- `RoomAnswer` — ответ участника на вопрос.

Такой набор сущностей позволяет покрыть все требования ТЗ: авторизацию, создание викторин, запуск комнаты, ответы игроков, подсчёт очков и хранение истории.

## 6. Реализованная функциональность
В рамках MVP реализованы следующие возможности:
- регистрация и авторизация пользователей с ролями `игрок` и `организатор`;
- создание викторины организатором;
- редактирование названия, описания, категории и списка вопросов;
- поддержка вопросов с одиночным и множественным выбором;
- добавление изображений к вопросам;
- запуск комнаты по короткому коду;
- подключение игроков к комнате по коду;
- показ активного вопроса всем участникам в одно и то же время;
- ограничение ответа временем показа вопроса;
- автоматический подсчёт баллов;
- обновление лидерборда по ходу игры;
- сохранение результатов в историю игрока и организатора.

## 7. Проверка и тестирование
В процессе работы были выполнены:
- проверка TypeScript для клиента и сервера;
- production-сборка клиента и сервера;
- проверка чистого запуска через Docker;
- проверка сценария реального времени в нескольких клиентских сессиях с ролями организатора и игрока;
- тестирование сценария:
  - вход организатора;
  - создание комнаты;
  - подключение игрока по коду;
  - запуск вопроса;
  - отправка ответа;
  - обновление лидерборда;
  - завершение игры;
  - сохранение результата в истории.

Также проверялось, что проект поднимается на пустой базе данных после `docker compose up --build`.

## 8. Результат
Итогом работы стал работоспособный прототип веб-приложения для проведения квизов в реальном времени. Проект соответствует основным требованиям ТЗ и может использоваться как демонстрационный MVP.

Пользователь получает:
- готовый интерфейс для игроков и организаторов;
- серверную часть с БД;
- синхронную игровую механику через WebSocket;
- запуск через Docker без сложной ручной настройки.
